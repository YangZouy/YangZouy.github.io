一文多发产品调研

产品定位与核心价值

1. 产品定位

面向知识型长文创作者的一文多发 Agent，输入 word 笔记，内容增强、自动适配不同平台并自动分发。

2. 目标用户

用户 A:技术博主

  • 背景:自建站个人博客
  • 内容类型:技术教程、项目复盘
  • 发布频率:每周 2-3 篇
  • 平台:个人博客 + 掘金 + 公众号
  • 痛点:内容需要复制粘贴到不同平台,且图片处理方式不同,博客需要外链,公众号可以直接上传平台图床。
  • 使用场景:周末集中写作,希望一次写作多平台分发

用户 B:转行者

  • 背景:互联网开发者,正在转型 AI PM
  • 内容类型:产品思考、行业分析、转行进程汇报
  • 发布频率:每周 1-2 篇
  • 平台:公众号 + 小红书 + 知乎
  • 痛点:公众号排版丑,小红书需要配图和标题党
  • 使用场景:工作日晚间写作,希望快速发布

3. 核心价值

提升效率,自动进行文章编排,创作者专注于内容创作,对于各个平台,省去粘贴复制文章到编辑器的时间,省去上传本地图片到图床的时间,agent 直接进行自动发布。

痛点与用户规模

痛点 频率 强度 优先级
个人知识库(One Note )文章转换到不同平台的文章格式 每次发布 高 P0
多平台登录+复制粘贴 每次发布 高 P0
图片如何放入到图床的方式 个人知识库发布时 中 P0
小红书需要重写标题 每次发布 高 P1
文章排版调整 + 概要生成 + 封面图生成 每次发布 高 P1

用户规模：数据来源分散,规模分析不需要 100%精确,采用分层漏斗+交叉验证的估算思路,保证逻辑严谨,后续进行或删除。

深度分析

1. Wechatsync

试用报告:

  • 选用文章一:博客优化二。(先在个人博客中进行发布)
    将已经上传的个人博客内容同步到上述各个平台,过程如下图:

点击上图中的调整按钮,出现编辑器页面,插件不能获取到博客封面图,直接使用文章内容第一张插件作为封面图,正文内容和标题都可以直接修改,但仅限文字。对应平台:文档存在草稿中,并未发布,可在平台上进行内容审核修改。

其中人人都是产品经理、掘金、语雀、知乎的图片样式丢失,例如使用 html 标签插入图片的大小和显示位置改变,通过它们各自提供的编辑器,可以进行一定的修改进行发布。微信公众平台对于文档内容的编排不够好。

小红书


小红书对于知识型长文来说不合适,且 markdown 格式也不适用于该平台,其自有编辑器包含丰富的图文素材和模板。

  • 选用文章二:给 n8n 的本地访问链接配置成公网访问(先在公众号进行发布)
    个人博客:hexo 搭建的是静态站点,需要将公众号的文章通过下载的方式将网页转成markdown,插件将文章标题、正文、封面图智能提取出来,将内容插图保存在压缩包中,用户需自行推送到个人博客站点。其中 markdown 文件缺少 front-matter 部分,需自行补充(个人博客 markdown 格式需要有该部分,主要内容为标题、日期、分类、标签、背景图、封面图、概要),且链接失效。
    缺点:未发布前无法在 md 预览时看到图片状态,因为图片嵌入位置是按照发布后的位置填写的。
    扩展:在静态资源处理上,静态站点与动态站点之间的区别是不同的。
    知乎、掘金、语雀、人人都是产品经理、小红书与之前效果相同。该插件可以从浏览器的任意文档型页面进行同步。

工作流程

技术原理

  • 内容提取：浏览器插件扫描网页,通过多算法评分机制识别正文区域,择优提取。
  • 格式适配：提取到的 HTML 经清洗并转换为 Markdown 后,由平台适配器(Adapter)按需输出HTML 或 Markdown,以适配不同平台。
  • 请求注入：从浏览器存储中读取目标平台的 Cookie 与 CSRF Token,构造 JSON 数据包,通过POST 请求发送至平台草稿保存,完成内容同步。

选型启发

该产品本质是提效类工具,不涉及多账户托管,产品形态应尽量简单轻便,功能明确,降低使用门槛。所以插件类和 agent 类更适用于个人 cope 和开发者,但插件类受限于浏览器环境的限制,且很多平台并未给出官方注入 API,内容注入时 API 发生变化,插件代码需要修改,进行频繁维护。且功能中需要有内容排版、封面图生成等功能,需要使用到大模型的能力。
